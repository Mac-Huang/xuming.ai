Where does Intel NPU compile time go?

Compiling a model for the Intel NPU under OpenVINO can take minutes — long enough to block real work. This decomposes a cold NPU compile into its MLIR compiler passes to find what actually dominates, how it scales with model size, and what is worth optimizing.

Every number is measured on the real int8 OpenVINO IR on a fanless Intel Core Ultra 7 356H mini-PC, by parsing the NPU compiler's own per-pass timing report — so it profiles the exact compiler you run, no source build required. Models: Qwen2.5-1.5B / 7B (LLM, NPUW_LLM window-swept) and ViT-base / large (vision) for the modality contrast.

Key findings

  • NPU compilation is genuinely expensive — and it scales with the model. A cold compile runs 15–28 s for Qwen2.5-1.5B and 82–242 s for Qwen2.5-7B (int8, NPUW_LLM). One compile is actually 7 sub-model compiles under the hood.
  • Almost all of it is codegen the blob cache removes. Warm reuse of a cached blob compiles in ~0.7–1.9 s — up to 125× faster than cold. The whole cost is one-time codegen; the irreducible floor is a <2 s blob load.
  • What dominates depends on model size, not the “interesting” passes. For the 1.5B the leader is MultiClusterStrategyAssignment (tiling/multi-cluster strategy). For the 7B it flips to FeasibleAllocation — the memory allocator + scheduler — which alone is 72 s (~31% of the whole compile), and the alloc+schedule stage reaches 40%.
  • The allocator is the super-linear wall. Whole-compile time grows only ~window^0.24 for the 1.5B but window^0.55 for the 7B, and the driver is alloc+schedule, which scales ~window^1.2 (super-linear). That is the pass to attack.
  • ELF backend / serialization is a fixed ~20–24% floor on every model, small or large, vision or language — a surprisingly large share for what is “just” writing the blob (ConvertVPUMI40XX2VPUASM and friends).
  • No source build needed. The installed release compiler already prints a full per-pass duration report at INFO level (IE_NPU_LOG_FILTER=vpux-compiler) — so this profiles the exact compiler you run. On the pip OpenVINO 2026.0 wheel, default and DRIVER are the same compiler-in-driver; there is no compiler-in-plugin.

2 · What dominates a cold compile?

The pipeline splits into a small set of stages. The striking result: the dominant stage shifts with model size. For the 1.5B, tiling + multi-cluster strategy search leads; for the 7B, the memory allocator/scheduler (FeasibleAllocation) explodes to dominate.

The 7B's single most expensive pass is FeasibleAllocation — the memory allocation + scheduling pass — at tens of seconds on its own. It is also the only stage that grows super-linearly with window, which is exactly why the 7B compile balloons.

3 · Does the model change the bottleneck? Size, not modality.

Profiling a vision transformer (ViT) against the LLM across sizes shows there is no “vision compiles differently” story. A ViT and an LLM of similar compile cost have nearly the same pass breakdown; what changes the bottleneck is scale. Small models (ViT-base, Qwen-1.5B) lead with multi-cluster strategy; large ones (ViT-large, Qwen-7B) tip toward FeasibleAllocation. ELF-serialization stays a ~20–27% floor throughout.

The lever is FeasibleAllocation + caching. The scheduler dominates large-model compiles and is super-linear, so approximating/parallelizing it (or bounding the window) is the highest-value target. Operationally, robust blob caching keyed to (model, window) already turns a 4-minute 7B compile into a sub-2-second load — the single biggest practical win.

Method & reproducibility

The installed compiler prints a per-pass duration report at LOG_INFO when OV_NPU_LOG_FILTER=vpux-compiler is set before import openvino; the harness captures the native stderr (fd-level), parses the MLIR PassTiming tree, and buckets each pass into pipeline stages. Every compile runs as an isolated timeout-guarded subprocess (the NPU compiler crashes non-deterministically). Cold = NPU_BYPASS_UMD_CACHING=YES; warm = an OpenVINO CACHE_DIR. Env /data/envs/openvino-3.12 (OpenVINO 2026.0.0), NPU driver-compiler 1.33. Code: experiments/npu_compile/ in the edge-ai repo (microbench.py · stages.py · analyze.py). Run folders are write-once. A white-box cross-check builds vpux-opt --mlir-timing from openvinotoolkit/npu_compiler.
