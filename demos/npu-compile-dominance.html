Does FeasibleAllocation actually dominate NPU compile time?

A controlled sweep on Intel NPU 5 (OpenVINO 2026.0) across Qwen2 (dense attn), ViT (vision encoder) · 36 cold-compile configurations · per-pass timing from the VPUX compiler's own report.

Short answer: no — not in general.

FeasibleAllocation (the memory allocator + list-scheduler) is the single heaviest pass in only 8 of 36 configurations. In 16 of 36 the heaviest pass is MultiClusterStrategyAssignment — the tiling / multi-cluster strategy search. Across the whole sweep FA ranges just 2.45%–33.36% of pass time. It only takes over at large model × large window: it needs ~7B params and a long prompt window before it dominates (reaching ~33% of the whole compile, ~44% at the stage level). And it's a hump, not a staircase — it peaks at W1024 then declines as the window grows to the 32K context limit, handing the compile back to ELF serialization.

3 regimes: tiling → scheduling → serialization FA is #1 only ≥7B, W512–16384 FA peaks ~33% @W1024, falls to ~20% @W32768 ELF-serialize retakes #1 at W≥16384 FA flat ~5–8% for ≤3B at every window to 32K

The crossover: which stage owns the compile

Ordering every configuration by cold compile time (cheap → expensive) shows a clean hand-off. Tiling + strategy search owns the cheap compiles (~24–34%) and shrinks with scale; memory allocation + scheduling starts at ~11% and climbs to ~44% on the most expensive ones. ELF serialization is a steady floor throughout. FeasibleAllocation isn't a fixed bottleneck — it's the pass whose cost scales.

Three regimes, not one bottleneck (window swept to the 32K context limit)

Sweeping the prompt window across its full range, W = 128 … 32768 (Qwen2.5's native context), the dominant pass moves through three regimes as the compile grows:

  1. Small models / short windows → tiling-search. `MultiClusterStrategyAssignment` is the heaviest pass; FA is a flat ~5–8% and never crosses over — true for 0.5–3B at every window up to 32K (ViT sits here too; it's size, not modality, that matters).
  2. 7B, mid windows → memory scheduling. On the 7B, FA climbs 5% → 10% → 33% as the window goes 512 → 1024 (it schedules every tiled sub-op through the tiny CMX, and the op count grows with size×window). This is the FA-dominant band, W512–~8192.
  3. Extreme windows → serialization. Push further and FA peaks at W1024 (~33%) then declines — 31 → 32 → 28 → 27 → 20% at W2048/4096/8192/16384/32768 — because its scheduling cost saturates against fixed CMX while the compiled artifact (DMA descriptors, barriers, weight tables) grows linearly with the window. So ELF serialization / lowering takes the share back: on the 0.5–3B models the #1 pass at W16384–32768 is already ConvertVPUMI40XX2VPUASM or DispatchedInliner, not a search or a schedule.

So FeasibleAllocation isn't a fixed bottleneck or an unbounded blow-up — it's a middle-window hump, gated by model size, bracketed by tiling-search below and serialization above. Memory stayed bounded throughout (peak <1 GB even at 7B/W32768 — NPUW chunks the prefill), and the 7B compile time grew only ~230 → 370 s from W1024 to W32768.

Which architectures even compile on NPU 5?

Before profiling, each family has to compile at all — and that turns out to be a real filter. We tried five architecture families; only dense-attention decoders and vision encoders compile. Both sparse-MoE and hybrid-SSM models fail, and for the same underlying reason: they emit data-dependent dynamic shapes (variable tokens-per-expert; SSM state) that the NPU's fully-static compiler can't lower.

So the dominance results above are, necessarily, the story for the architectures the NPU can run — and that class is narrower than the model zoo suggests.

What this says to optimize

The bottleneck is regime-specific, so the lever depends on your workload:

  • Small/mid models (≤3B, any window): the target is the tiling / multi-cluster strategy search — and it runs fully serial ( MultiClusterStrategyAssignment user/wall = 1.0×), so it won't get faster on more cores. Caching or pruning the strategy search is the lever.
  • Large models, mid windows (≥7B, W≈512–8192): the target is FeasibleAllocation — a greedy simulate-and-spill list scheduler that re-runs 2–3× and is only mildly parallel. Bounding the window or shipping precompiled blobs keyed by (model, W) avoids it entirely.
  • Extreme windows (W≥16384) and every compile's floor: ELF serialization / materialization (ConvertVPUMI40XX2VPUASM, lowering) is a ~20–28% floor that grows with window until it retakes #1 — the artifact scales with W. It's mechanical, so the cheapest win is not recompiling at all (warm blob cache), and long-context deployments should ship precompiled ELF rather than pay materialization each time.
