Does NPU/CPU-assisted prefill make Holo1.5-7B faster and cheaper on the NUC 16?

A one-shot microbenchmark + OSWorld pilot for Holo1.5-7B (Qwen2.5-VL-7B) on a fanless Intel Core Ultra mini-PC, measuring where each pipeline stage should run — and whether the memory system, not compute, is the real bottleneck.

One representative OSWorld step: a 262144px screenshot → 1248 patches / 312 vision tokens, prompt length S=347 tokens. Every number below is measured on the real int8 OpenVINO IR with RAPL energy (reps-slope, net-of-idle); config totals compose the measured stages plus a unified-memory cross-device transfer tax (36.722 pJ/byte, measured).

Key findings

  • Verdict: NPU/CPU-assisted prefill does NOT help. The whole-iGPU baseline G0.1 whole-iGPU wins on both latency (8.522 s/step) and energy (169.67 J/step).
  • The NPU cannot execute the batched prefill graph at all (the T=S compile is rejected/hangs) — so every NPU-prefill placement falls back to token-by-token prefill.
  • That makes each NPU-prefill config (G0.2 / G0.3 / H0) ~12× slower and multiples more energy than whole-iGPU — the compute-bound prefill belongs on the iGPU matrix engine, not the low-bandwidth NPU.
  • Memory is the decode bottleneck. Prefill on the iGPU is compute-bound (2.2 pJ/FLOP); decode is memory-bound (165.8 pJ/FLOP, ~75× worse) at 57.7 GB/s = 64% of DDR5-5600 peak. No accelerator fixes this — the fix is fewer weight bytes / token.
  • Success is not the discriminator. On the one-per-domain OSWorld subset the runnable placements score G0.3 0/10, G0.1 0/10 — so NPU placement does not reduce success (there is none to reduce). The deployment ceiling here is grounding + agent-loop robustness and the decode memory wall, not where prefill runs.

The five placements

Holo1.5's LLM is a plain transformer (KV cache only, 3-section M-RoPE — no Gated-DeltaNet), so it runs coherently on the NPU directly and prefill/decode can be split across devices with a zero-copy KV handover.

Where each stage belongs

Vision and prefill are compute-heavy (the accelerators help); decode is a per-token full-weight read (the accelerator barely matters — the memory system does).

The memory wall

Each decode token streams the whole ~7 GB int8 weight set from DRAM, so decode runs near the DRAM roofline regardless of device. Prefill amortizes that same weight read across all S context tokens in one batched pass — which is why prefill is compute-bound (~2 pJ/FLOP on the iGPU) and decode is memory-bound (~100–170 pJ/FLOP). The intuition motivating this study was that the compute-bound prefill might do better on the NPU's matrix engine; the measurement says otherwise.

Why the NPU loses. Two independent reasons, both measured: (1) the NPU cannot compile the batched (T=S) prefill graph — it gives up after retries — so "prefill on NPU" degrades to a token-by-token loop of S sequential steps; and (2) even per token, the NPU streams weights at only ~27 GB/s (30% of DDR5 peak) versus the iGPU's ~58 GB/s, so its decode is ~2× slower. The NPU is a low-power, low-bandwidth engine; it sips energy per token but is the wrong home for either the compute-bound prefill or the bandwidth-bound decode here.

How to actually optimize the memory side. Because decode is bounded by weight bytes moved per token, the levers are all on the bytes, not the device: coarser weight quantization (INT4/ mixed), weight streaming/residency so the read is not re-paid every token, or amortizing one weight pass over more tokens (speculative / multi-token decode). Relocating decode to another accelerator does not help — all three are within a factor of the same DRAM roofline.

H1 — fine-grained operator split

H1 would put the big matmuls (QKV/O projections + FFN) on the NPU and keep attention/norm/RoPE + decode on the iGPU. But since the NPU cannot compile the prefill graph, its per-op prefill cost is unavailable, and the conservative composition collapses H1 back onto the iGPU (≈ G0.1) — the matmuls have no faster home. A finer probe that compiles a single QKV/FFN op in isolation on the NPU is future work, but it would still have to beat the per-layer NPU↔iGPU activation-shuffle tax that Phase 1 prices in.

Phase 2 — OSWorld pilot (success preservation)

One task per domain (round-robin, 8 steps max) on the runnable placements. The question was whether NPU-assisted prefill reduces success. It does not — because success is 0/10 either way on this hard subset. Inspecting trajectories, Holo1.5-7B forms coherent intents and roughly-correct grounding, but a single-image history (max_images=1) makes it repeat a click when a slightly-off target leaves the screen unchanged, and it occasionally emits out-of-range coordinates. The whole-NPU arm additionally can't reach the resolution its grounding needs (NPU vision compiler ceiling ~262144 px). Neither limit is about prefill placement — so the offload question is moot for success, and the latency/energy verdict above stands on its own.

Method & reproducibility

Model Hcompany/Holo1.5-7B → OpenVINO int8 stateless IR (162/162 LLM layers int8_sym). Energy via powercap RAPL (no root), reps-slope net-of-idle marginal per-infer energy. Serial, quiet machine. Code: experiments/holo15/ in the edge-ai repo (microbench.py · analyze.py · serve.py). Run folder is write-once.
