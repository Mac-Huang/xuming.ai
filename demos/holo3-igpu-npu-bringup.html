Holo3 serves fine on the NUC's CPU (llama.cpp, 4-bit GGUF). Getting it onto the Xe3 iGPU or the NPU was the hard part. This is the honest story in four parts — why it failed, what we tried, where we landed, and what to try next.

1 · Why it wouldn't run

Holo3 is a qwen3_5_moe vision-language model — an unusual mix that today's accelerator toolchains don't fully support yet:

  • Linear-attention layers (Gated-DeltaNet, an SSM-style scan) instead of plain attention.
  • A 256-expert Mixture-of-Experts — only 8 experts fire per token (3B of 35B params active).
  • A built-in vision encoder (ViT) fused into the model.
  • We only hold it as a GGUF file (llama.cpp's format), not the safetensors the Intel tools expect.

There are two ways to reach the accelerators, and each hits a different wall:

DevicePathWhat blocks it
iGPUllama.cpp (Vulkan) The LLM offloads fine, but the vision part crashes the GPU (vk::DeviceLostError).
OpenVINO The exporter now exists (we fixed it — see below) and the ViT already runs on the iGPU. But the 35B LLM won't export: the converter's tracer rejects its data-dependent branching, and even if it didn't, building the graph needs ≥94 GB RAM.
NPUOpenVINO only The NPU needs fixed shapes and no dynamic branching — but the MoE routing and the SSM scan are both dynamic. It also tops out around 8B; the LLM is 35B.

In plain terms: the CPU works because llama.cpp already implements this whole architecture. The accelerators are catching up — the vision half is now unblocked on both; the 35B language half is the stubborn part.

2 · What we tried

AttemptResult
LLM on the iGPU (llama.cpp Vulkan) ✓ Works and is correct. June's "garbage output" bug did not come back (a driver update fixed it). ~1.6× faster prefill.
Vision on the iGPU (llama.cpp Vulkan) ✗ Still crashes even on the newest build — an unresolved bug in llama.cpp's vision path on Intel.
OpenVINO exporter for the arch ✓ Broke the tooling deadlock with a community fork (rkazants) + newer transformers. A small qwen3_5_moe exports and runs on CPU + iGPU — so the exporter itself is real.
Whole 35B → OpenVINO (the goal) ✗ Two walls. The converter's tracer rejects the LLM's data-dependent branches (the mask update + the image-token position logic) — it dies before touching memory. And the graph build needs ≥94 GB RAM (measured: OOM-killed at 94 GB, still climbing). We tried disk-offload + 71 GB swap — no help, because the wall is tracing, not RAM.
Vision on OpenVINO (CPU/iGPU/NPU) ✓ Exported the real vision tower (weights pulled from the GGUF), verified bit-exact vs the reference (cosine 1.0). Runs everywhere — iGPU is fastest (~0.4 s vs ~5 s on CPU).
LLM on the NPU ◑ Rewrote it in a static, dense form (fixed shapes, run all experts + mask, unrolled scan). A tiny version compiled and ran on the NPU (cosine 0.9998) — proof it's possible. But dense-MoE costs ~32× at the real 256/8 ratio, and the NPU caps ~8B, so the full model still won't fit.
OSWorld on the iGPU (quick check) ◑ Quality fine (7/10 at a short step cap). Speed was a wash — each step is dominated by the vision encode, which is stuck on the CPU.

3 · Where we are now

Each piece of the model now has a device it runs on correctly:

StageRuns best onStatus
Vision encode (ViT)iGPU via OpenVINO (or NPU)✓ working, bit-exact, iGPU fastest
LLM prefilliGPU via llama.cpp✓ ~1.6× faster than CPU
LLM decodeCPU✓ CPU still wins single-stream decode
Whole model, end-to-endCPU via llama.cpp✓ the only full path today — served on :8082

What we achieved. The vision encoder is fully unblocked on the iGPU (OpenVINO, proven correct), the LLM runs correctly on the iGPU (llama.cpp), and we proved the NPU can execute this architecture (tiny model). Both walls that looked impossible — the OpenVINO exporter and the NPU — turned out to be crossable.

What can be done right now. A hybrid runner — LLM on the iGPU (llama.cpp) + vision on the iGPU (OpenVINO) in one server — which removes the CPU vision bottleneck that made OSWorld a wash. That is the whole pipeline on the iGPU today, without a full OpenVINO export.

What still can't be done here. Running the whole model as one pure-OpenVINO graph. We freed the box (116 GB disk, 89 GB RAM) and gave it 71 GB of swap, but the 35B export dies at the tracer (data-dependent control flow) before memory even matters — and behind that sits the ≥94 GB graph-build wall. Neither is fixable by freeing more RAM.

4 · What to try next

  1. Make the LLM trace-safe (the real unblock). Patch the handful of data-dependent branches in the model's forward — the mask update and the image-token position logic — into static / trace-safe forms. This is the same trick that got a static-dense qwen3_5_moe onto the NPU. Fix that and the fork's exporter should go through.
  2. Then export on a bigger box. Even once it traces, the graph build needs ≥94 GB RAM (offload/swap can't help — the graph must be fully materialized). Export on a ≥128 GB machine, then copy the ~18 GB 4-bit IR back — inference fits the NUC fine.
  3. Ship the hybrid runner (works today). LLM on the iGPU (llama.cpp) + vision on the iGPU (OpenVINO) in one server — the whole pipeline on the iGPU without the full export, and it fixes the real OSWorld bottleneck (vision off the CPU).
  4. Keep the NUC as the model server. Benchmark runs (OSWorld, etc.) move to the Legion box over the network; the NUC just serves Holo3.
