1 · Why the 35B wouldn't run

Holo3-35B is a qwen3_5_moe vision-language model — an unusual mix that today's accelerator toolchains don't fully support:

  • Linear-attention layers (Gated-DeltaNet, an SSM-style scan) instead of plain attention.
  • A 256-expert Mixture-of-Experts — only 8 experts fire per token (3B of 35B params active).
  • A built-in vision encoder (ViT) fused into the model.

There are two ways to reach the accelerators, and each hits a different wall:

DevicePathWhat blocks the 35B
iGPUllama.cpp (Vulkan) The LLM offloads fine, but the vision part crashes the GPU — I reproduced it exactly this round: vk::DeviceLostError inside clip_image_batch_encode, and it's size-triggered (a small image encodes fine; a real screenshot loses the device).
OpenVINO The exporter exists and the ViT runs on the iGPU, but the 35B LLM won't export: the converter's tracer rejects its data-dependent branching, and even past that the graph build needs ≥94 GB RAM.
NPUOpenVINO only The NPU wants fixed shapes and no dynamic branching — but the MoE routing and the SSM scan are both dynamic, and it tops out around 8B while the LLM is 35B.

In plain terms: the CPU works because llama.cpp implements this whole architecture. On the accelerators the vision half is fine; the 35B language half is the stubborn part — the MoE and the sheer size are what push the export past the memory wall.

What I achieved. The whole 9B runs on the iGPU as one OpenVINO graph — the thing the 35B could never do — and it drove a full OSWorld episode to a correct result on this machine: no Legion box, no remoting. The vision half also runs on the NPU (verified correct). The three walls that looked fatal on the 35B — the Vulkan ViT crash, the OpenVINO export, the NPU — are all crossed on the 9B, or crossed for the half that matters.

How the 9B run is wired. A small OpenAI-compatible server wraps the OpenVINO model on the iGPU (:8091); the OSWorld agent talks to it exactly like it did the CPU server. Two serving details matter: the turn-stop token had to be corrected (<|im_end|>, or it runs away), and image size is capped — the OpenVINO GPU plugin throws CL_OUT_OF_RESOURCES on a big prefill, so I bound the screenshot to ~1 280×720 and keep one image in context.

The honest limits. Decode on the iGPU is ~3 tok/s — the Gated-DeltaNet layers aren't well-optimized in the OpenVINO GPU plugin — so a full 369-task OSWorld sweep is days, not hours. And the context grows ~170 tokens/step, so long episodes eventually approach the same GPU memory ceiling. The 35B still has no pure-OpenVINO path (RAM + tracer).

Why the NPU can't run the LLM (yet) — it's not an unsupported op. The natural guess is that the Gated-DeltaNet has some exotic operation the NPU lacks. It doesn't: I asked the NPU compiler to vote on every node, and all 7,450 ops in the LLM are individually supported. It still fails to compile.

The real reason is that the NPU is a static dataflow engine: it wants the whole computation as one fixed-shape graph with no runtime branching and no mutable memory. The ViT is exactly that — a pure feed-forward pass — so it compiles and runs (verified). The LLM is not: it is stateful. It carries a KV cache (for the full-attention layers) and a recurrent SSM state (for the Gated-DeltaNet layers) as mutable internal memory — ReadValue/Assign nodes that persist and update across tokens. Even after I pin every tensor to a fixed shape, that persistent mutable state is something the NPU compiler can't assemble into its static graph. So it dies at the whole-graph level, not on any single op.

The fix — and it works. Make the model stateless: lift the KV and SSM state out of the graph and pass them as explicit inputs and outputs, so one step becomes a pure function (state_in, tokens) → (logits, state_out) at fixed shapes. Then the NPU sees a static feed-forward graph with no hidden memory. Re-exporting the 9B with stateful=False does exactly this — the KV and the Gated-DeltaNet conv/SSM states become explicit static inputs — and the full 9B LLM then compiles and runs on the NPU (174 s to compile). The graph-level wall is crossed.

5 · What to try next

  1. Speed up 9B decode on the iGPU. ~4 tok/s is the bottleneck (prefill is a healthy ~200 tok/s). Try int4 weights, and profile the Gated-DeltaNet ops in the OpenVINO GPU plugin — the SSM scan / conv1d are the likely culprits.
  2. Run the whole 9B on the NPU. The stateless LLM now compiles on the NPU and the vision tower already runs there — the remaining work is wiring a stateless decode loop (feed the KV/SSM state in and out each step) plus a static prefill graph, so the entire model runs on the NPU. Then compare NPU decode against the iGPU's ~4 tok/s.
  3. Split the pipeline across accelerators. ViT on the NPU, LLM on the iGPU (or vice-versa) — each device does the half it's best at, in one live server.
  4. Revisit the 35B only if needed. If the 35B's quality is required, the trace-safe patch + a ≥128 GB export box, or the llama.cpp + OpenVINO hybrid, remain the two routes.
