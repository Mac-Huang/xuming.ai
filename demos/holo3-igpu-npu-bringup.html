The problem

Holo3-35B-A3B serves fine on the NUC's CPU (llama.cpp, Q4_K_M) — but every attempt to run it on the Xe3 iGPU or the NPU 5 (AI Boost) had failed: garbage output on full GPU offload, and no OpenVINO path at all for the NPU. This page states why (it is an architecture problem, not a driver problem), the plan to fix each accelerator, and the measured result of two focused bring-up agents — one per device.

One-line diagnosis. Holo3 is a qwen3_5_moe vision-language model: a hybrid Gated-DeltaNet (linear-attention) + 256-expert MoE with an integrated ViT, and it ships to us only as a GGUF. That combination is friendly to llama.cpp and hostile to OpenVINO — which is exactly why the CPU works, the iGPU only works through llama.cpp's Vulkan backend (with one real bug), and the NPU can host only the vision encoder, never the language model.

Outcome. After two focused bring-up agents: the LLM now runs correctly on the iGPU (~1.6× prefill, ~1.45× end-to-end — June's output-head "garbage" did not reproduce), and the vision encoder runs on the NPU (0.32–6.0 s static f16, fastest device at the merged shape). Remaining hard limits: vision-on-GPU crashes the Vulkan device, and the LLM on the NPU is architecturally impossible — now measured, not assumed.

1 · Why it wouldn't run — five stacked reasons

Holo3-35B-A3B is Qwen3_5MoeForConditionalGeneration (model_type: qwen3_5_moe, transformers 5.2.0). Its 40 decoder layers alternate 3× linear-attention (Gated-DeltaNet / SSM-style: chunked scan + causal conv1d) with 1× full-attention (full_attention_interval=4 → only 10 KV-bearing layers), each MLP is a 256-expert MoE, 8 active (35B total / 3B active), and a 27-layer SigLIP-class ViT is fused in. The two accelerator toolchains react very differently:

#BlockerHitsConsequence
1GGUF-only on disk — no safetensorsOpenVINO (iGPU+NPU) optimum-intel cannot convert a GGUF → no IR
2No qwen3_5_moe converter; env has transformers 4.55.4 < required 5.2.0OpenVINO (iGPU+NPU) arch cannot even be instantiated to trace
3Gated-DeltaNet linear attention (SSM scan + conv1d)OpenVINO / NPU no clean lowering to ONNX / OV ops
4256-expert dynamic MoE routing (gather/scatter)NPU (static-shape) dynamic dispatch is hostile to a static compiler
5Dynamic-dim norms / scan rejected by the vpux compilerNPU requires fully static shapes

So the picture inverts per device. CPU: llama.cpp already implements the whole arch from the GGUF → works. iGPU: reachable only through llama.cpp's Vulkan backend (OpenVINO is out for reasons 1–3), and there it hit a real correctness bug on full offload. NPU: reachable only through OpenVINO, so the language model is categorically out (reasons 1–5) — the one Holo3 component that can map to it is the vision encoder, and only with static shapes.

2 · The plan — one agent per accelerator

Because the two devices fail for different reasons, the fix split cleanly into two independent bring-up agents, run in parallel on the live machine. Each had to resolve an open question empirically, not just re-state the theory.

▸ Agent A — iGPU (Intel Xe3, llama.cpp Vulkan) goal: a correct, useful GPU config

The repo's own records contradicted each other: the June notes said full offload -ngl 99 produced garbage (output-head/embedding offload bug, upstream #21888), while the GPU serve script claimed -ngl 99 gave a 2.5× prefill win — but that number was throughput only, never correctness-checked, and the on-disk Vulkan binary is the stale June build. Agent A had to settle it on the current build and produce a config that is both correct and useful.

  1. Correctness probe: greedy, deterministic decode at -ngl 0 / 40 / 99 — find where output turns to garbage.
  2. Throughput: llama-bench prefill (pp512/2048) & decode (tg32), GPU vs CPU.
  3. The fix: pin the output head + token-embeddings to CPU (--override-tensor) while offloading all 40 blocks + experts to the Xe3; verify coherence returns and the prefill win survives. Fallback: rebuild Vulkan at a newer commit.

▸ Agent B — NPU (Intel AI Boost, OpenVINO) goal: run whatever can run

The language model is categorically un-exportable to the NPU (reasons 1–5). So Agent B's job was to prove that boundary concretely and then deliver the part that does map — the vision encoder as a static-shape OpenVINO graph on the NPU.

  1. Confirm LLM-on-NPU is infeasible: capture the exact optimum-intel/OpenVINO export failure.
  2. Deliver the ViT encoder on the NPU: static-shape IR (dynamic norms are rejected by vpux), verify it compiles + runs on the NPU today, measure latency NPU vs iGPU vs CPU.
  3. Document the proxy-weight caveat and the concrete path to real-weight bit-parity.

3 · Result — Agent A (iGPU)  LLM now runs correctly

Re-running the probes on the current Vulkan build settled the contradiction. Text now decodes correctly at full offload (-ngl 99) — the earlier "garbage" did not reproduce (the binary is unchanged, so the likely cause is a Mesa driver update since early June). The prefill win is real (~1.5–1.7×), decode is modestly slower (bandwidth-bound), and full multimodal serving works with the vision encoder on CPU — the one hard bug is that offloading the vision projector to the GPU crashes it.

Prefill is compute-bound → the Xe3 wins ~1.6×. Decode is bandwidth-bound → the 16 CPU cores + DDR5-5600 win batch-1 by ~16%. OSWorld serving is ~89% prefill, so the net is ~1.45× end-to-end.

The remaining bug. Offloading the vision projector to the iGPU (--mmproj-offload) loses the Vulkan device (vk::DeviceLostError, core dump). So Holo3's images cannot be encoded on the Xe3. The clean fix is to keep vision encode off the GPU — on the CPU, or (better) on the NPU, which is exactly what Agent B delivers.

Winning config (correct + useful). llama-server … --device Vulkan0 -ngl 99 --no-mmproj-offload -fa on -c 32768 --jinja — all 40 blocks + 256-expert MoE + output head on the Xe3, vision projector on CPU. Full multimodal serving works (it correctly reads a screenshot): ~1.5–1.7× prefill, ~1.45× end-to-end for the prefill-bound agent loop. One-line fix to the parked GPU serve script: --mmproj-offload → --no-mmproj-offload.

At the merged shape the NPU is fastest (0.32 s, 17× CPU). At full-res the iGPU leads on wall-clock; the NPU's edge is perf/watt and freeing the 16 CPU cores. iGPU 8120 today read 23 s under concurrent Vulkan bench contention → June's clean 3.3 s cited; NPU 8120 reused from June (fresh 39-min compile skipped). CPU 37 s brackets the real ~36 s CLIP encode → proxy validated.

Two costs to know. (1) Static shapes are mandatory — dynamic-dim LayerNorm/MVN is rejected by the vpux compiler, so the graph is reshaped to a fixed [1,N,1152]. (2) The one-time compile is slow — ~6 min at 2048 patches, ~39 min at full-res (CPU/GPU compile in <5 s). It is an offline, amortizable cost per input resolution.

Bit-parity is in reach. The real Holo3 vision tower is already on disk inside mmproj-f16.gguf (all 27 v.blk.* + the mm.0→GELU→mm.2 merger, dims matching the proxy exactly). Latency is shape/op-bound, not weight-bound, so today's numbers stand; wiring the real weights into the OV graph for output-parity is the tracked next step.

5 · What runs where, today

The two bring-ups compose into a clean per-stage placement. The GPU can't touch vision (it crashes) but the NPU can; the GPU wins prefill; the CPU still wins batch-1 decode. Holo3 is a three-accelerator model.

Bottom line. Holo3 does run on the NUC's accelerators once you place each stage where it maps: LLM on the iGPU at -ngl 99 -fa (correct output, ~1.45× end-to-end on the prefill-bound agent loop) with the vision encoder on the NPU as a static f16 graph (fastest at the merged shape; frees the CPU; best perf/watt). What genuinely cannot run on an accelerator is the language model on the NPU — a hard architecture limit (qwen3_5_moe: dynamic 256-expert MoE + Gated-DeltaNet SSM vs. a static-shape compiler with no exporter), not a bug to be fixed. That boundary is now measured, not assumed.

Open items (see §6 for the 2026-07-07 fix attempts): (1) the --mmproj-offload Vulkan device-lost crash — confirmed unresolved on llama.cpp master, keep vision off the GPU meanwhile; (2) real-weight NPU vision parity (weights already local in the mmproj GGUF); (3) a fused three-device serving path (NPU encode → iGPU prefill → CPU decode) in one runner.

6 · Can the two hard blockers be fixed? — attempted 2026-07-07

Two follow-up bring-up agents tried to actually repair the two remaining walls — the vision-on-GPU crash and the missing OpenVINO exporter — on the current toolchain (network available; latest packages). Both were pushed to a definitive, reproducible outcome. Neither is fixable on this box today; here is exactly why, and the precise unblock condition for each.

▸ Vision-projector GPU crash  not fixed upstream

Built llama.cpp at upstream master (bec4772, a month past our da87e9b) in an isolated worktree and re-tested --mmproj-offload with real screenshots. The crash still fires: mtmd_batch_encode: vk::Device::waitForFences: ErrorDeviceLost → terminate, reproduced on 2 of 3 runs; the one run that survived returned a hallucination (called an Ubuntu desktop "a website … guide to Ubuntu"), i.e. the crash is merely traded for silently corrupted vision embeddings. Control on the same prompt (read the top-bar clock, truth 09:13): vision-on-CPU + LLM-on-GPU answers "09:13"; full-CPU answers "09:13"; vision-on-GPU crashes. Verdict: an unresolved upstream llama.cpp Vulkan bug for the Holo3 (qwen3_5_moe) CLIP path on Intel Xe3 — a rebuild does not fix it. Working config stands: -ngl 99 --no-mmproj-offload (LLM incl. output head on the iGPU, vision encode on CPU — or on the NPU).

▸ qwen3_5_moe → OpenVINO exporter  arch recognizedexport blocked

Stood up a fresh env with the newest packages. Good news: transformers 5.13 now recognizes qwen3_5_moe (arch + modeling class present) — the "unrecognized architecture" half is real-fixable by upgrading transformers. Bad news: the only OpenVINO exporter, optimum-intel 2.0.0 (released and git-main), hard-pins transformers<5.1 — which does not know the arch (KeyError: qwen3_5_moe) — and it fails to even import under transformers 5.13 (API-incompatible). So no installable combination both recognizes the arch and loads the exporter: a dependency deadlock. Export to OpenVINO IR is not runnable today; unblock condition = optimum-intel shipping transformers-5.2+ support. And two walls remain even after that: the NPU has no kernels for the Gated-DeltaNet / 256-expert MoE family (documented CPU&GPU-only; NPU is static-shape, caps ~8B — the 35B LLM cannot run on it regardless), and a full 35B export is disk-gated (~70 GB safetensors + ~70 GB IR > 123 GB free). The already-working iGPU path for the Holo3 LLM is llama.cpp Vulkan, not OpenVINO.

Net. Neither blocker is a config mistake — both are upstream gaps with named unblock conditions: a llama.cpp Vulkan CLIP fix, and optimum-intel transformers-5.2+ support. The NPU-for-LLM boundary is a genuine hardware/vendor limit, not a bug. Everything that does work today (LLM on iGPU via llama.cpp, vision on NPU via OpenVINO, decode on CPU) is unchanged and correct.
