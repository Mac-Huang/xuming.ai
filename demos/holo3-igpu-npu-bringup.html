1 · Why the 35B wouldn't run

Holo3-35B is a qwen3_5_moe vision-language model — an unusual mix that today's accelerator toolchains don't fully support:

  • Linear-attention layers (Gated-DeltaNet, an SSM-style scan) instead of plain attention.
  • A 256-expert Mixture-of-Experts — only 8 experts fire per token (3B of 35B params active).
  • A built-in vision encoder (ViT) fused into the model.

There are two ways to reach the accelerators, and each hits a different wall:

DevicePathWhat blocks the 35B
iGPUOpenVINO The exporter exists and the ViT runs on the iGPU, but the 35B LLM won't export: the converter's tracer rejects its data-dependent branching, and even past that the graph build needs ≥94 GB RAM.
llama.cpp (Vulkan) The LLM offloads fine, but the vision part crashes the GPU — I reproduced it exactly this round: vk::DeviceLostError inside clip_image_batch_encode, and it's size-triggered (a small image encodes fine; a real screenshot loses the device).
NPUOpenVINO only The NPU wants fixed shapes and no dynamic branching — but the MoE routing and the SSM scan are both dynamic, and it tops out around 8B while the LLM is 35B.

In plain terms: the CPU works because llama.cpp implements this whole architecture. On the accelerators the vision half is fine; the 35B language half is the stubborn part — the MoE and the sheer size are what push the export past the memory wall.

▸ On the NPU (NPU 5 · Intel AI Boost)

AttemptResult
35B (qwen3_5_moe)
35B · Vision / LLM / whole ✗ out of scope for the NPU: it caps ~8B and needs fixed shapes + no dynamic branching, but the MoE routing and the SSM scan are both dynamic. (A tiny static-dense proxy did compile — proof it's possible — but the full 35B doesn't fit.)
4B / 9B (qwen3_5 dense) — the real problem: make the LLM static
Vision tower + token embeddings ✓ compile and run on the NPU with fixed shapes — the ViT matches CPU to cosine 0.999995 (~22 ms).
LLM — as exported (stateful) ✗ compile fails at the graph level, not on any op: all 7,450 ops are individually supported, but the mutable KV + Gated-DeltaNet SSM/conv state (ReadValue/Assign) won't assemble into the NPU's static dataflow.
LLM — re-exported stateless (the fix — partial) ◑ stateful=False lifts the KV and the SSM/conv state out as explicit static inputs, and the LLM decode runs on the NPU (static T=1 loop): 4B "the capital of France is Paris" at 6.4 step/s, 9B at 3.9. But only decode. A batched prefill graph (T>1) will not compile on the NPU — the multi-token Gated-DeltaNet scan is rejected, exactly like the stateful graph (tried chunk sizes 64–512, contexts 512–2048; all fail). So a real prompt is prefilled either token-by-token on the NPU (~0.18 s/token → ~205 s for a 1.1k-token OSWorld prompt) or on the iGPU — hence partial, not a full-NPU win. And there's a second wall: even token-by-token, the NPU output goes incoherent past ~250 tokens and reverts to echoing the prompt. It is not precision (the iGPU runs the same f16 and stays correct to 1.5k) and not a bug in my loop — I root-caused it to a specific NPU op-lowering fault in the Gated-DeltaNet layers (sweep + op-level diff below).
Image→text — full VLM step (vision iGPU → LLM NPU) ✓ a real screenshot → the LLM decodes the answer on the NPU: 4B "Go to DataAccess (unsafe)", 9B "the app is requesting access to sensitive Google-account info" — correct grounding. (The vision runs on the iGPU; the NPU ViT is static-shape and would need manual orchestration to join the loop.)
OSWorld served on the NPU ✗ I ran the full one-per-domain 4B subset on the NPU (10 tasks, ~15 h) → 0/10. The pipeline runs end-to-end, but at OSWorld prompt lengths the decode is incoherent: every step it echoes the tool schema / prior screenshot instead of emitting an action, so all 10 tasks fall back to WAIT. I confirmed the cause with an iGPU-vs-NPU sweep — the same token-by-token loop and the same IR, only the device changes:
prompt tokens → does it still answer "capital of France?"
12 · 250 · 488 · 998 · 1508   iGPU ✓ ✓ ✓ ✓ ✓  ·  NPU ~ ✗ ✗ ✗ ✗ (NPU echoes the prompt from ~250 tokens on; iGPU is correct to 1.5k)
I then chased it all the way down, and it is an Intel NPU-compiler miscompute — not precision, not the weights, and not any single op. Ruled out one by one: precision (CPU forced to f16 is coherent, and both devices default to f16); weights (bf16 and true fp16 fail identically). A per-layer differential shows the very first GDN layer is bit-perfect on the NPU (conv & SSM state cos 1.0) and divergence only appears at layer 1 and compounds through depth. Every distinctive op is NPU-exact in isolation — the delta-rule SSM cell, the causal conv1d, and the gated RMSNorm all match CPU to cosine 1.0. And the two structural oddities I could remove — the once-only scan Loop (inlined away) and a shape-derived NonZero varlen gather (constant-folded away) — changed nothing. No compiler-backend toggle helps (DRIVER≡PLUGIN; ACCURACY mode makes it worse — NaN). So: every op is right alone, the same graph is right on CPU-f16 and the iGPU, but the NPU's fusion/lowering of the full 32-layer Gated-DeltaNet decode graph is wrong — a compiler defect, not something fixable from user space. Verdict: the NPU can't run this architecture correctly today; the iGPU (whole-model, 4B 7/10) is the path, and this is a clean OpenVINO NPU bug to file. The 9B subset never ran — a stray 93 GB export OOM-killed the overnight compile.

5 · What to try next

  1. Speed up 9B decode on the iGPU. ~4 tok/s is the bottleneck (prefill is a healthy ~200 tok/s). Try int4 weights, and profile the Gated-DeltaNet ops in the OpenVINO GPU plugin — the SSM scan / conv1d are the likely culprits.
  2. File the NPU-compiler bug; don't expect a user-space NPU fix for this architecture. NPU-served OSWorld is blocked by an Intel NPU-compiler miscompute of the full Gated-DeltaNet decode graph, which I chased to the bottom: it is not precision (CPU forced to f16 is coherent), not the weights, and not any single op — the first GDN layer is bit-perfect on the NPU, and every distinctive op (SSM cell, conv1d, gated RMSNorm) is NPU-exact in isolation (cos 1.0). Removing the once-only Loop and the shape-derived NonZero varlen gather changed nothing, and no compiler-backend toggle helps. The same graph is correct on CPU-f16 and the iGPU, so the NPU's fusion/lowering of the 32-layer graph is at fault. The right move is a minimal-repro OpenVINO NPU bug report (isolated ops pass, full graph fails); until Intel fixes it, the iGPU is the whole-model path. (Also still open: the T>1 prefill graph won't compile on the NPU at all, so even a fixed decode would need token-by-token prefill or an iGPU prefill hand-off.)
  3. Move the vision onto the NPU too. The ViT compiles on the NPU but the OV model's vision sub-models are dynamic-shaped — reshape them static and orchestrate the merger's rotary/mask from grid_thw to run the whole VLM (vision + LLM) on the NPU.
  4. Split the pipeline across accelerators. ViT on the NPU, LLM on the iGPU (or vice-versa) — each device does the half it's best at, in one live server.
  5. Revisit the 35B only if needed. If the 35B's quality is required, the trace-safe patch + a ≥128 GB export box, or the llama.cpp + OpenVINO hybrid, remain the two routes.
