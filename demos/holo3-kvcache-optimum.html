The goal

Today we trade one good thing for another. Evict (keep the last 3 screenshots) keeps context short and quality high — but rewriting an old screenshot in place breaks the KV prefix every step, so the whole history is re-prefilled (~0% cache hit, 126 s prefill/step, quadratic on long tasks). No-evict (keep everything) makes the prefix stable so the cache is reused (~2× less prefill) — but context grows unbounded: decode slows, memory OOMs, and accuracy drops −11 pp. The optimum we want is both at once: keep only the last few screenshots (short context → full quality, fast decode, no OOM) and make every step a cache hit on the retained history (only the new visual content is re-prefilled). This page is the build guide — and the honest account of what the hybrid architecture lets us actually reach.

The answer, in one line. The naive "evict from the middle of the KV and renumber positions" is unsound on Holo3 (30 of its 40 layers are recurrent — no per-token KV to splice). The exact, buildable optimum is terminal screenshot slots + a state checkpoint/restore: park the text history in a checkpoint, re-prefill only the current ≤3-image window each step. It is exact for all 40 layers and the primitive already ships in llama.cpp (--ctx-checkpoints).

How to achieve it — the mechanism & build

Build guide and research roadmap for bounded-context, full-quality, cache-resident acceleration of Holo3-35B-A3B on llama.cpp (CPU). This page assumes the architecture facts established in holo3-osworld-acceleration.html; read it first for the hybrid-layer analysis and the ruling-out of --cache-reuse.

1. The goal, and the core idea in one paragraph

Goal. Today we are forced to choose between two bad arms:

  • evict (H Company default, keep last 3 screenshots by trimming the prompt): bounded context → full quality and fast decode, but every step diverges the token sequence near the trimmed image, so the server re-tokenizes and re-prefills everything after the first divergence — KV-cache hit ≈ 0%, prefill ≈ 126 s/step.
  • no-evict (append-only, keep all screenshots): the stable prefix is a true cache hit so prefill drops to ≈ 63 s/step (KV hit ≈ 62%), but context grows without bound → decode slows as context grows, the quality A/B drops −11 pp, and the tail OOMs.

We want both: keep the context bounded to ≤ K screenshots (the model's training regime → full quality, fast decode, no OOM) and make the stable text history a true cache hit so only the current screenshot window is prefilled.

Core idea. Lay the prompt out so all screenshots sit at the end — [preamble][append-only TEXT action history][≤ K current screenshots] — and take an exact state checkpoint at the image-region start (the end of the stable text prefix, before the screenshots). Strict causality makes that checkpoint an exact summary of the prefix [0..p] for all 40 layers (the 10 attention layers' KV and the 30 Gated-DeltaNet recurrent states). Each step we restore the checkpoint, prefill only the current ≤ K-screenshot window (a constant ~6k tokens for K=3, never growing), decode the action, append the new action text to the stable prefix, and advance the checkpoint by that small text delta. The text history is restored, never recomputed (a true cache hit); only the screenshots are re-encoded. This is the bounded, exact, full-quality optimum on this hybrid model.

2. Why the naive "evict from the middle of the KV and renumber" is unsound here

The intuitive optimum — drop the oldest screenshot's KV span from the middle, renumber the survivors, and re-rotate — is the seq_rm + seq_add + K-shift triad (--cache-reuse / context-shift). On a plain transformer with per-token KV this is exactly what StreamingLLM (Xiao et al., ICLR 2024) does: it assigns positions within the cache rather than from source text and re-applies RoPE to the survivors. Block-Attention (Ma et al., ICLR 2025) gives the de-rotate → zero → re-rotate-to-target math; CachedAttention/AttentionStore (Gao et al., USENIX ATC 2024) is the production analogue ("positional-encoding-decoupled truncation"). All of these assume per-token KV and therefore map only onto the 10 attention layers of Holo3. They are silent on the other 30.

On this model the triad is unsound for two independent reasons:

(a) --cache-reuse is inert here anyway — it is disabled for multimodal. In llama-server, cache-reuse is turned off for multimodal contexts (server-context.cpp:997) and refused per-request whenever the prompt carries image tokens (can_cache_reuse = can_shift && !has_mtmd, server-context.cpp:2716). Every Holo3 prompt carries image tokens, so the path never even fires. The OpenAI /v1 endpoint exposes none of this regardless.

(b) Even if you forced it, the K-shift only touches 10 of 40 layers — and the other 30 cannot be spliced. Holo3-35B-A3B (qwen3_5_moe) is a hybrid: only 10 layers are full-attention with a position-indexed KV cache; the other 30 are Gated-DeltaNet / linear-attention layers carrying an O(1) recurrent state (one running summary per sequence, s_t = s_{t-1}·g_t + k_t·d_t), not per-token KV. A K-shift re-rotates cached keys — it has no meaning for a recurrent state. And llama_memory_seq_add on a recurrent memory only bumps a single tail cell (llama-memory-recurrent.cpp:304); it cannot reconstruct a gapped-prefix running state. Editing any token at position p invalidates every recurrent state after p, and there is no per-token KV to splice back in. So seq_rm+seq_add+K-shift would leave 10 layers approximately-correct and silently corrupt 30/40 layers.

This is the deep reason the evict arm measures 0% KV hit and 126 s prefill while no-evict gets 62% / 63 s: prompt-level trimming forces a full re-prefill, and there is no sound in-cache shortcut for this architecture. The fix is not "shift the KV better" — it is to never put a screenshot in a position we will later have to remove from the middle.

The importance-based eviction literature (H2O, Scissorhands, SnapKV, GUI-KV, STaR-KV, VL-Cache) answers which tokens to drop, and GUI-KV/STaR-KV even report accuracy gains over full-cache on multi-screenshot GUI histories. But every one of them selects at the attention-score layer and deliberately does not touch positions or recurrent state — they presuppose the per-token-KV substrate we only partially have. They are useful for choosing K and which screenshots to keep, not for keeping the retained context a cache hit on this hybrid model.

3. The correct mechanism on this llama.cpp: terminal slots + checkpoint/restore

The architecture forbids middle-splice but permits an exact terminal operation. Strict causality cuts both ways: a state snapshot taken at position p is an exact summary of the prefix [0..p] for all 40 layers — attention KV and recurrent ssm/conv state. So if every screenshot lives strictly after p, we never edit anything before p, the snapshot stays valid forever, and we only ever recompute the bounded suffix.

3.1 Prompt layout (terminal slots)

position 0 ──────────────────────────────────────────────► n
┌───────────┬──────────────────────────────┬───────────────────────────┐
│ preamble  │ append-only TEXT action hist │  ≤ K current screenshots  │
│ (system,  │  step1 action, step2 action, │  <observation>img…</obs>  │
│  task)    │  …  (grows by ~text/step)    │  (the ONLY image region)  │
└───────────┴──────────────────────────────┴───────────────────────────┘
            ▲                              ▲
       checkpoint A0                  checkpoint Ck  ← "image-region start"
       (after preamble)               (end of stable text prefix,
                                        BEFORE any screenshot)

This is the Prompt Cache (Gim et al., MLSys 2024) "modular slot" discipline made architecture-honest: the screenshots are a single relocatable terminal region, and the checkpoint is the schema-defined boundary between the stable prefix and that region. ChunkAttention's two-phase (shared-prefix / fresh-suffix) kernel is the exact compute pattern — a long reused span plus a small fresh span.

Take the checkpoint Ck at the image-region start — the end of the stable text prefix, immediately before the first screenshot token. That is the largest prefix that is stable across the step (the action text for the step we are about to take has not been generated yet), so it maximizes what is reused.

3.2 Per-step algorithm

state:  Ck            # checkpoint at end of stable text prefix (all 40 layers)
        prefix_text   # preamble + action history (the tokens Ck summarizes)
        window        # deque of ≤ K most recent screenshots (image-token spans)

loop each agent step:
    1. new_shot = capture_screenshot()
       window.push(new_shot)
       if len(window) > K: window.pop_oldest()      # bounded → training regime

    2. restore(seq=0, Ck)                            # exact prefix summary, all 40 layers
       # cache now holds [0..p] for attention KV AND recurrent state — a true hit,
       # nothing in prefix_text is recomputed.

    3. prefill(window)                               # ONLY the ≤K-image window
       # ~6k tokens for K=3, CONSTANT. re-encode the screenshots; this is the
       # only recompute. recurrent state advances forward from Ck over the window.

    4. action = decode_action()                      # bounded context → fast decode

    5. prefix_text += render_action_text(action)     # append-only, small delta
       restore(seq=0, Ck)                            # back to the boundary…
       prefill(action_text_delta)                    # …extend by the text delta only
       Ck = checkpoint(seq=0, at end of prefix_text) # ADVANCE checkpoint

    # next step re-prefills only the next ≤K window; text history is never recomputed.

Step 5 is the key bookkeeping move: the screenshot window is transient (recomputed each step), but the action text is durable, so we fold only the new action text into the checkpoint. The checkpoint walks forward over the stable prefix one small text delta at a time; the screenshots never enter it.

Cost: per-step prefill = |window| ≈ 6k (constant), not the growing 8k–27k re-prefill we pay today. The text history is a restore (cache hit), not a recompute.

3.3 The exact llama.cpp primitives — this already ships

No model change. This is agent + server orchestration on top of primitives that are in the tree today:

  • --ctx-checkpoints (on by default, n=32): the server's checkpoint ring. This is the feature to repurpose — instead of opportunistic checkpoints, place one deterministically at the image-region boundary and advance it each step.
  • llama_state_seq_get_data_ext(... LLAMA_STATE_SEQ_FLAGS_PARTIAL_ONLY) → serialize a sequence's state at the current position (our checkpoint(Ck)).
  • llama_state_seq_set_data_ext(... LLAMA_STATE_SEQ_FLAGS_PARTIAL_ONLY) → restore it (our restore(Ck)).
  • Crucially, these serialize both the attention KV and the recurrent state in one blob (server-context.cpp:2033; recurrent path llama-memory-recurrent.cpp:864). That is precisely why this is sound where K-shift is not: the checkpoint is an exact, whole-model prefix summary, including the 30 Gated-DeltaNet layers that cannot be shifted. CachedAttention's session save/restore (USENIX ATC 2024) is the published analogue, and Mooncake (FAST 2025) / LMCache (2025) are the tiering tier if these blobs need to be parked rather than held — LMCache independently measured the pathology we are fixing: production context-truncation halves the prefix-cache hit ratio (~85% → ~45%).

4. Correctness and quality — honest treatment

This scheme is exact for all 40 layers. A state snapshot at position p is the model's true internal state for the prefix [0..p] — there is no approximation in the attention KV and none in the recurrent state. Restoring it and continuing is bit-identical to having computed the whole prefix. This is the decisive difference from the K-shift family, which is approximate even on the 10 attention layers (Block-Attention needs a fine-tune to recover accuracy; EPIC/CacheBlend/KVLink all add selective recompute or link tokens to repair relocation error) and undefined on the 30 recurrent layers. We pay zero correction budget because we never relocate anything.

The one remaining open question is not numerical — it is layout. The terminal-slot prompt reorders the conversation relative to the interleaved leaderboard layout (where each screenshot sits next to the action it produced). The model still sees the same information, but image observations are now clustered at the tail and older observations survive only as text. The empirical question is whether that reordering changes agent behavior.

What to test (accuracy A/B, holding everything else fixed):

  1. Terminal-slot layout vs. interleaved layout, both at K=3 screenshots, on the same OSWorld-Verified subset (the same clean n≈46 used for the evict/no-evict A/B). Primary metric: task score. This isolates the only thing the optimization changes.
  2. Sensitivity to K (K=2,3,4): confirm K=3 stays in Holo3's training regime and that going terminal does not shift the sweet spot.
  3. Text-history fidelity: ablate what of each old step survives as text (full action JSON vs. summarized) — terminal layout pushes more weight onto the text history, so verify the action text carries enough state.
  4. Exactness sanity check: assert restore-then-prefill produces token-for-token identical logits to a from-scratch full prefill on a fixed prompt (numerical proof the checkpoint is exact on this hybrid model end-to-end).

If (1) shows parity, the optimum is fully realized. If the reorder costs a little, FastV's "visual info migrates to text in early layers" result (Chen et al., ECCV 2024) and the GUI-KV/STaR-KV redundancy findings suggest the cost is small and recoverable by tuning what text survives — but that is a layout tweak, not a correctness fix.

5. The purest ideal — and why it is architecturally impossible to do bounded here

The theoretically purest step would be ~2k tokens/step: re-encode only the new screenshot, with every retained token (text and the older retained screenshots) a cache hit, and context bounded. VLCache (Qin et al., Alibaba/Qwen, 2025) is almost exactly this for VLMs on the same model family (Qwen2.5-VL/Qwen3-VL, including the 30B-A3B class): store image KV pre-RoPE as a hash-keyed slot, re-apply position at the destination, recompute 2–5% of tokens.

It is impossible to do bounded on this hybrid model, because keeping the older retained screenshots as cache hits while bounding context requires splicing an old screenshot out of the middle of the sequence — and that middle-splice is exactly what the 30 Gated-DeltaNet recurrent layers forbid (Section 2). VLCache, like all of Section 2's prior art, presupposes per-token KV; it has no recurrent state to reconstruct. There is no pre-RoPE relocatable slot for a running summary.

The append-only / no-evict arm does already realize the ~2k/step prefill (re-encode only the new screenshot, everything else a hit) — but only by leaving context unbounded. That is the trade it cannot escape: it buys minimal prefill with decode slowdown, the −11 pp quality A/B, and an OOM tail. You can have minimal prefill (no-evict, unbounded) or bounded exact (terminal-slot checkpoint), but not the pure bounded + minimal-prefill ideal — the architecture removes that option.

Terminal-slot checkpoint/restore is therefore the optimum that actually exists for this model: it trades a constant ~6k/step (re-encode the K-image window) to keep the model in its ≤ K-screenshot training regime, and wins decisively over both evict (re-prefill everything, quadratic in steps) and no-evict (unbounded, slow decode, OOM) on long, high-step tasks.

Research & practical roadmap

The mechanism is settled in the sibling page /demos/holo3-osworld-acceleration.html (§2, §2b): on this hybrid qwen3_5_moe model — 10 full-attention layers with a position-indexed KV cache, 30 Gated-DeltaNet layers carrying an O(1) recurrent state — the literal "swap a screenshot out of the middle of the KV" move (llama_memory_seq_rm + seq_add + K-shift, i.e. --cache-reuse) is both inert (disabled for multimodal prompts) and unsound (it would silently corrupt 30 of 40 layers, since the recurrent state cannot be position-shifted). The bounded-context-and-cache-hit optimum is reached instead by terminal slots + exact state checkpoint/restore: lay the prompt as [preamble][append-only TEXT action history][≤K screenshots at the END], checkpoint the full state (attention KV and recurrent ssm/conv state) at the image-region start, and each step restore the checkpoint and re-prefill only the current ≤K-screenshot window (~6 k tokens, constant). This section maps the prior art onto that mechanism, ranks the practical paths, gives a validation plan, and renders a verdict.

Read this first: almost every result below was developed on standard transformers with per-token KV on all layers. On those models, the "relocate/evict a KV segment in the middle and re-rotate positions" family is sound. On Holo3 it is sound only for the 10 attention layers — the 30 recurrent layers have no per-token KV to splice, so the exact analogue of any "KV segment" trick is a state checkpoint, not a KV edit. Keep that asterisk on everything in Clusters 1–4.

1 · Related work, mapped to our problem

Cluster A — Modular / position-independent KV reuse (segment / slot caching)

This is the cluster the user half-remembers, and it is the conceptual ancestor of "a screenshot is a reusable slot." The MLSys work is Prompt Cache (Gim et al., MLSys 2024): it defines reusable prompt modules with an explicit schema that assigns each module a position range in the new prompt, so a cached segment can be slotted in at a different offset than where it was computed (up to 8× lower TTFT on GPU, 60× on CPU, no quality loss). The screenshot analogy is exact at the abstraction level — each screenshot is a modular KV slot with an assigned [p0,p1) position range, and we track that span from the mtmd injection. The key limitation here is architectural: Prompt Cache assumes the segment is per-token KV that can be re-slotted by re-assigning positions. On our 30 recurrent layers there is no per-token KV to re-slot — the exact summary of a prefix is the recurrent state at that position, so the faithful realization of a "reusable screenshot slot" is a state checkpoint of the prefix that ends where the screenshots begin, not a relocatable KV chunk. That is precisely why our slots must be terminal (§2b of the sibling page).

The rest of the cluster supplies the segment-relocation machinery that is valid only for our 10 attention layers: - Block-Attention (Ma, Wang, Tian, ICLR 2025) is the closest published analogue to an M-RoPE K-shift: it caches each block at RoPE-position 0, then de-rotates → zeroes → re-rotates the cached KV to its true global position (98.7 % TTFT reduction, ≤1 % RAG gap after a block fine-tune). It gives us the exact angle arithmetic for shifting the attention half — but it is 1-D-RoPE text only and needs a fine-tune we cannot apply to a fixed Q4_K_M model. - VLCache (Qin et al., Qwen Team / Alibaba, arXiv Dec 2025) is the multimodal realization, evaluated on the same family as Holo3 (Qwen3-VL-30B-A3B): store image KV pre-RoPE, hash-key by pixel content, re-apply position at the destination, recompute only 2–5 % of vision tokens (1.2–16× TTFT, integrated in SGLang). It is the most transplantable template — but it reuses identical recurring images and lives above libllama in SGLang, and it still presumes per-token KV on every layer. On the recurrent half its "store pre-RoPE, relocate, recompute 2–5 %" recipe has no defined meaning. - CacheBlend (Yao et al., EuroSys 2025, Best Paper) and EPIC (Hu et al., ICML 2025) supply the correction primitive for relocation error on attention layers: recompute only the high-deviation tokens (CacheBlend, ~15 %) or just the first ≤32 boundary tokens of a relocated chunk to kill the false attention sink (EPIC/LegoLink). KVLink (Yang et al., NeurIPS 2025) couples position re-index + learned "link" tokens. These are the right tools if we ever relocate an attention-layer segment — but they require a fine-tune (KVLink) or assume 1-D RoPE text (all four), and none has a recurrent-state story. ChunkAttention (Ye et al., ACL Findings 2024) gives the data-structure / kernel pattern (chunked KV in a prefix tree, two-phase partition for "long reused span + small fresh span") — directly the compute shape of "restore prefix + prefill only the new screenshots."

Mapping verdict for Cluster A: the segment/slot abstraction is right; the implementation (relocatable KV chunk + RoPE re-rotation + small recompute) is sound only for our 10 attention layers. The 30 recurrent layers force the exact analogue to be a state checkpoint, which is what makes terminal-slot checkpoint-restore — not a KV-pointer swap — the correct mechanism.

Cluster B — Streaming / sliding-window KV + attention sinks + position renumbering

This cluster legitimizes "renumber surviving positions inside the cache." StreamingLLM (Xiao et al., ICLR 2024) is the canonical result: keep ~4 attention-sink tokens + a rolling recent window, and assign positions within the cache, not from the source text (re-applying RoPE to surviving keys) — 22.2× over sliding-window recompute, stable to 4 M tokens. LM-Infinite (Han et al., NAACL 2024, Outstanding) adds the distance-limit principle (keep relative positions inside the trained range), and Self-Extend (Jin et al., ICML 2024, Spotlight) proves position ids can be rewritten inference-only (floor-division remap) with no retraining. The production realization of all three is llama.cpp context-shift / --cache-reuse (the seq_rm + seq_add + K-shift triad, n_keep as the sink anchor), and PR #13870 ("llama : fix KV shift for qwen2vl") even makes the K-shift M-RoPE-aware by rotating the temporal component as NEOX whole-vector (θ=−1).

Mapping verdict for Cluster B: every primitive here edits per-token KV positions. On Holo3 that path is the one the sibling page ruled out — it touches only the 10 attention layers and silently corrupts the 30 recurrent ones (the recurrent seq_add only bumps a single tail cell). What this cluster does give terminal-slot checkpoint-restore is two transferable disciplines: (1) pin the sink — keep the instruction/preamble prefix stable (it is the checkpoint root), and (2) the distance-limit intuition — because the ≤K screenshots sit at the end and the text history is append-only, retained positions never leave the trained range, which is the whole reason the terminal layout is benign. DuoAttention (Xiao et al., ICLR 2025), H2O (Zhang et al., NeurIPS 2023) and VL-Cache inform which content is safe to bound, not how to keep it a hit.

Cluster C — Importance-based KV eviction / compression (which tokens to keep)

This cluster answers a question terminal-slot checkpoint-restore mostly dissolves: with the ≤3 most recent screenshots always re-prefilled at the end, we never have to choose which old screenshot KV to drop — we keep a bounded window by construction. Its value is at the margins: GUI-KV (Huang et al., Salesforce, arXiv 2025) and ST-Lite (Zhou et al., arXiv 2026) are screenshot-trajectory-native (spatial L2-saliency + cross-frame temporal redundancy; UI-element-aware grouping), and GUI-KV reports +4.1 pp step accuracy at −38.9 % FLOPs on a 5-screenshot context — evidence that bounded screenshot context can raise accuracy, directly supporting our "bounded ≠ worse" claim. STaR-KV (Han et al., arXiv 2026) is the most recent successor (+0.38 pp vs full cache at a 40 % budget). VL-Cache (Tu et al., ICLR 2025) gives the modality-aware, layer-adaptive budget rule (image KV is far more compressible than text). Quest (Tang et al., ICML 2024) is the conceptual pivot for the cache-HIT constraint: it never evicts — it keeps full KV resident and gates which pages enter attention by query-aware min/max bounds, so the prefix stays bit-identical. H2O / Scissorhands / SnapKV / FastGen supply the text-LLM importance signals.

Mapping verdict for Cluster C: these choose what to drop; terminal-slot checkpoint-restore needs no victim selection because the window is fixed at the end. They become relevant only if we want K > 3 (use GUI-KV/ST-Lite redundancy scoring to thin the extra screenshots) or want to push the bound further on the attention layers only (VL-Cache budgets, Quest page-gating — neither touches the recurrent state, so both are safe to layer on top of checkpoint-restore but neither replaces it).

Cluster D — Serving systems: paged / prefix / block KV + reuse infra

The production substrate. PagedAttention/vLLM (Kwon et al., SOSP 2023) gives block-table indirection (the structure that makes any "free one screenshot's blocks" expressible); RadixAttention/SGLang (Zheng et al., NeurIPS 2024) gives the radix-tree prefix index + LRU. The decisive, independently-measured fact comes from LMCache (Liu et al., arXiv 2025): production context truncation drops the prefix-cache hit ratio from ~85 % to ~45 % on real traces — production-scale corroboration of our measured 62 %→0 % collapse, and the motivation for not truncating the prompt. CachedAttention/AttentionStore (Gao et al., USENIX ATC 2024) is the closest serving analogue of our mechanism: a positional-encoding-decoupled KV save/restore that survives a window slide and re-embeds positions on load (87 % TTFT reduction) — the session-granularity ancestor of checkpoint-restore. CacheBlend reappears here as the non-prefix correction pass; Mooncake (Qin et al., USENIX FAST 2025, Best Paper) is the tiered KV-pool worldview.

Mapping verdict for Cluster D: all of these treat KV as the first-class object — exactly the worldview in which "checkpoint this prefix, restore it next step" is a scheduling decision. But every one is prefix/block/text oriented; none models M-RoPE image positions or, critically, a Gated-DeltaNet recurrent state. vLLM/SGLang would hit the same hybrid-recurrent constraint we hit, and they currently lack mtmd + Gated-DeltaNet checkpoint serialization — which llama.cpp already ships via llama_state_seq_get/set_data_ext(PARTIAL_ONLY).

Cluster E — Multimodal / VLM KV management + M-RoPE position correctness

The model-spec and risk cluster. Qwen2-VL (Wang et al., arXiv 2024) / Qwen3-VL Technical Report (arXiv 2025) define M-RoPE (temporal/height/width components; image tokens hold a constant temporal id + per-patch h/w) — this is Holo3's position spec. The primary llama.cpp artifacts PR #13870 (M-RoPE shift as NEOX whole-vector, θ=−1) and issue #17930 ("Qwen2.5-VL fails with M-RoPE sequence position error") confirm that position-relocating image spans is a fragile workaround that is still failing upstream — the open risk that, on this model, we avoid entirely by never relocating image positions (terminal slots + checkpoint-restore re-prefill the screenshots at their natural positions). FastV (Chen et al., ECCV 2024) and VL-Cache justify that image KV is largely droppable after early layers; Elastic Cache (Liu et al., ECCV 2024) offers anchor-merging as a softer alternative; StreamingLLM recurs as the renumbering precedent; UI-TARS (Qin et al., arXiv 2025) frames the workload (pure-screenshot OSWorld agent, OSWorld 24.6/22.7).

Mapping verdict for Cluster E: this cluster's headline gift is negative and clarifying — the M-RoPE-shift correctness risk that dominates Clusters A/B is designed out by the terminal-slot layout, because we never move an image token's position; we restore an exact prefix and re-encode the screenshots fresh. The GUI-agent works (UI-TARS, GUI-KV, STaR-KV) supply the evaluation framing and the "≤3 screenshots is near the training regime" justification.

3 · Experiment plan — validate the optimum on this setup

Instrumentation (per step, logged to the run folder): - prefill tokens processed (the headline — target: constant ~6 k for path 1; ~2 k for path 2; 8–27 k growing for the evict baseline) - KV-cache hit ratio (expect ~0 % evict, ~62 % no-evict, ~100 % on the text history for checkpoint-restore — only the ≤3 images are recomputed) - decode throughput (t/s) as a function of resident context length (catches the no-evict decode-slowdown tail) - task score (OSWorld-Verified success), per-step latency, wall-clock / task, energy (Wh) - peak KV memory (must stay bounded for path 1; must be watched for OOM on path 2) - checkpoint cost: state_seq_get/set serialize time + bytes (must be small vs the prefill it saves)

A/B arms (matched clean subset, n≈46, same seeds/tasks as the existing measured A/B): 1. Evict baseline — current max_images=3 prompt-level trimming (the 100 %-score, 126 s-prefill, 0 %-hit arm). 2. No-evict / append-only — the running unbounded arm (89 % score, 63 s-prefill, 62 %-hit, decode 0.89×). 3. Terminal-slot checkpoint-restore — path 1 (the candidate optimum).

Success criteria: - Quality: arm 3 matches arm 1 (evict) within noise on task score — i.e. recovers the −11 pp that arm 2 loses, by staying in the 3-image regime. - Speed/memory: arm 3 holds prefill ≈ constant ~6 k tokens/step while KV memory stays bounded (no growth, no OOM), and decode does not degrade with task length (unlike arm 2). - Net: arm 3 wins on wall-clock and energy vs arm 1 on long/high-step tasks (constant vs growing prefill), without arm 2's unbounded-context penalties.

Key failure signal: terminal-layout accuracy drift — if arm 3 (screenshots-at-END layout) scores below arm 1 (interleaved leaderboard layout) beyond noise, the regression is the prompt reordering, not the caching (checkpoint-restore is bit-exact). Diagnostic: run a layout-only control — interleaved vs terminal layout with identical no-cache prefill — to isolate reorder cost from checkpoint cost. Secondary signal: checkpoint serialize/restore time approaching the prefill it saves (would erase the win on a small box).

4 · Verdict

Yes — the theoretical optimum is reachable in practice on this exact stack, and it is the only one of the four paths that is. Terminal-slot checkpoint-restore is exact, not approximate: strict causality makes a state snapshot at position p a faithful summary of [0..p] for all 40 layers (attention KV and recurrent state), so it sidesteps both the M-RoPE position-shift risk (we never relocate an image token) and the recurrent-state corruption that kills --cache-reuse (we never splice the recurrent state — we replay it). Crucially, the primitive already ships — llama.cpp's --ctx-checkpoints with llama_state_seq_get/set_data_ext(PARTIAL_ONLY) serializes both halves of the hybrid memory — so this is agent + server orchestration, no model change (cross-ref /demos/holo3-osworld-acceleration.html §2b). The single highest-leverage first step: build the terminal-slot layout + checkpoint-restore loop in the agent harness and run the 3-arm A/B (evict / no-evict / checkpoint-restore) on the matched clean subset, with a layout-only control to separate prompt-reorder cost from caching — that one experiment confirms (or refutes) the whole scheme and tells us the true accuracy cost of moving the screenshots to the end. ```
